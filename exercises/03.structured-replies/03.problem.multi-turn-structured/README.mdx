# ЁЯТк Multi-turn structured replies

The comparison streams fast now, but the customer hates it: every follow-up question wipes the previous comparison off the screen. `partial` and `final` on `useChat` only ever track the *latest* in-flight reply, so once the new stream starts the old one is gone.

That's not a bug in your code тАФ it's the wrong tool. `partial`/`final` are designed for single-shot extractions (parse a `Person` out of an email, return a typed itinerary from one prompt). For a multi-turn chat, TanStack AI uses a **different mechanism**: every assistant message carries its own `structured-output` part with its own `partial`, `data`, and `status`. You walk `messages`, find each turn's part, and render it. Old cards stay rendered because they were never coming from the hook in the first place тАФ they're stored on their own assistant messages.

In this exercise you'll rewrite the chat component to use that pattern.

## ЁЯРи What you will do

One file тАФ `app/components/comparison-chat.tsx` тАФ and two small changes, both flagged by Kody ЁЯРи right where they happen:

1. **Read `messages` from `useChat`.** A `const messages = []` placeholder keeps the page rendering its empty state for now тАФ pull the real `messages` off the `useChat` destructure and delete the placeholder. (Last exercise you read `partial`/`final`; those only ever held the *latest* reply, which is why every follow-up wiped the previous comparison off the screen.)
2. **Render each turn.** Fill in the (currently empty) `messages.map` callback: return the right component for each message тАФ a user bubble for user turns, and for an assistant turn either an error card or the comparison card (pass `comparison ?? {}`, and the table renders skeletons until fields stream in).

## ЁЯжЙ Why each message owns its own part

In a single-shot extraction (`/api/extract-person`) the model writes one structured object and you're done тАФ there's nothing to keep history of, so `partial`/`final` on the hook are fine. The moment you put structured output inside a chat, every turn becomes a distinct, complete reply. Storing the typed result on the message itself is the natural fit: the conversation history *is* a list of structured replies.

The server side didn't change. The route is still `chat({ outputSchema: ComparisonSchema, stream: true })`. TanStack AI is doing the work of routing each assistant turn's JSON into its own `structured-output` part тАФ you're just reading from a different place.

## ЁЯжЙ The three states of `StructuredOutputPart`

```ts
status: 'streaming' | 'complete' | 'error'
```

- **`'streaming'`** тАФ `partial` is being updated, `data` is still undefined. Read `part.partial` to get a `DeepPartial<Comparison>`. The table already accepts this тАФ that was the previous exercise's work.
- **`'complete'`** тАФ `data` is set and is the validated `Comparison`. `partial` may still be present and equals `data` field-for-field; reading `data ?? partial` works either way.
- **`'error'`** тАФ validation failed. `errorMessage` describes why. Show a friendly error card; don't try to render the partial.

## тЬЕ How to know it is working

1. Make sure at least two products are in your comparison list.
2. Open `/compare` and send a question тАФ say, "what's the best value?". The first comparison streams in (skeletons тЖТ table).
3. Send a follow-up like "rank by durability instead." **The first comparison stays on screen**, the user bubble for the new question appears below it, and a second card streams in beneath that.
4. Scroll up тАФ your earlier comparison is still rendered exactly as it was when it finished.
5. Send a third question. A third card. Everything piles up in order; nothing is lost.
6. Refresh the page. History wipes (`useChat` doesn't persist across reloads тАФ that's a different problem) but the multi-turn rendering kicks in again as soon as you ask a new question.

## ЁЯЪи If you get stuck

- **TypeScript: `Property 'messages' does not exist` / `Cannot find name 'messages'`** тЖТ you deleted the `const messages = []` placeholder but didn't add `messages` to the `useChat` destructure.
- **A follow-up replaces the previous card instead of stacking** тЖТ make sure each card reads from *its own* message's `structured-output` part (`m.parts`), not from a single shared value.
- **Cards show up empty / show "тАФ" everywhere** тЖТ that's expected before tokens arrive: `comparison ?? {}` is `{}`, so the table renders skeletons until `part.partial`/`part.data` fills in.
- **TypeScript: `Property 'data' does not exist on type ...`** тЖТ make sure you're calling `m.parts.find((p) => p.type === 'structured-output')` directly inside the `messages.map` callback. The narrowing relies on `useChat`'s typed `messages` array.
- **A turn with `part.status === 'error'` crashes the table** тЖТ guard on `status === 'error'` *before* rendering the card; render `<ComparisonErrorCard message={part.errorMessage} />` instead.
